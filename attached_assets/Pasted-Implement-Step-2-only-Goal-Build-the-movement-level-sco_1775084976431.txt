Implement Step 2 only.

Goal
Build the movement-level scoring engine that sits directly on top of the existing Step 1 movement profile system.

Step 1 already exists and should remain the foundation:

canonical movement profiles
aliases
explicit muscle roles
15-pattern taxonomy
conservative cyclical movement handling
low-risk audit integration

Do not work on Steps 3–4.
Do not build prescribed vs performed architecture.
Do not build historical rollups.
Do not build readiness/recovery/fatigue logic.
Do not redesign the UI.
Do not add body map logic.

Step 2 objective
Given a single performed movement entry, return a weighted muscle-loading result.

The Step 2 engine should:

Resolve the canonical movement profile from Step 1
Calculate a V1 exposure score using available movement inputs
Convert the movement profile into a weighted muscle vector
Return pattern and stimulus summaries alongside the muscle vector
Stay heuristic, conservative, and physiologically believable
Be modular and easy to evolve later

Expected new modules
Please inspect the codebase and implement the cleanest version of the following modules, adjusting filenames only if there is a clearly better fit:

lib/movementScoringTypes.ts
lib/exposureScoring.ts
lib/muscleVector.ts
lib/stimulusVector.ts

Suggested types

PerformedMovementInput

Suggested responsibilities

exposureScoring.ts
Create a V1 heuristic scoring layer for:
loaded rep-based movements
bodyweight rep-based movements
duration-based holds
cyclical distance/calorie/time-based movements

Use conservative, explainable heuristics.
Do not pretend to have perfect sports-science precision.

muscleVector.ts
Build the weighted muscle vector:
resolve movement profile
get exposure score
multiply by base muscle weights
return per-muscle scores
stimulusVector.ts
Generate broad stimulus tendencies from the movement and exposure:
strength
hypertrophy
muscular_endurance
power
conditioning

This should remain simple and heuristic for V1.

Important guidance

Keep routes thin

Do not remove old logic broadly unless clearly safe
Keep the implementation portable and clean
Prefer pure functions and testable modules

Scoring guidance

Distinguish involvement from loading
A movement involving a muscle does not mean it got the same effective stimulus as the prime mover
Use movement type-aware heuristics
Keep cyclical movements conservative and distributed
Keep bodyweight skill movements differentiated by difficulty
Use available reps/load/duration/distance/calories when present
Return assumptions/metadata where useful
